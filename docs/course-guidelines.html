Instructor Onboarding:
Course Guidelines

Version 0.1

See http://authoring.datacamp.com/ for full documentation

Overview

  • Consistency reduces friction when learning
  • We want to implement good teaching practices
  • Our data tells us how to make online material more engaging

Foundations

Carroll et al: "The Minimal Manual". Human-Computer Interaction, 3(2), 1987

  1. Start learners on meaningful tasks immediately.
  2. Minimize reading and other passive activities.
    • Present instruction as self-contained single-page tasks.
    • Allow users to fill in gaps themselves.
  3. Include error recognition and recovery activities.

Orwell's Rules

  1. Never use a metaphor, simile or other figure of speech which you are used to seeing in print.
  2. Never use a long word where a short one will do.
  3. If it is possible to cut a word out, always cut it out.
  4. Never use the passive where you can use the active.
  5. Never use a foreign phrase, a scientific word, or a jargon word if you can think of an everyday English equivalent.
  6. Break any of these rules sooner than say anything barbarous.

Petzold's Rules

  1. Don't even try to compete with the docs.
  2. Forget about comprehensiveness.
  3. Write shorter chapters.
  4. It's all about the code.
  5. Get the reader involved.

Large-Scale Requirements

  • Most users should be able to complete a course in 2.5-3.5 hours
  • 4-5 chapters with 40-60 exercises in total
    • Chapter 1: 8-12 exercises (10 is best)
    • Other chapters: 10-16 exercises (15 is best)
  • First chapter is shorter because it's the funnel
    • And to give learners a sense of accomplishment

Voice

  • Inclusive voice
    • We are doing this together
    • "Let's remove Survived from the dataset."
  • Avoid exclamation marks!!
    • And remember that most jokes aren't funny the second time around

Exercises

  • Use coding exercises wherever possible
    • Pure MCQs only test recognition and recall
    • Unfortunately, today's technology can't grade free-text answers accurately
  • Length of instructions is the largest influence on completion rates
    • 1-4 instructions per exercise
    • Average 50 words per instruction
    • See documentation for up-to-date guidelines

Good Instructions

  • State the objective of the task
    • "Create a data frame called df containing 50 points on a circle and plot it."
    • Specify variable names to simplify checks
  • Make sub-goals explicit
    • "Create a vector of 50 angles from 0 to 2π radians."
    • "Display a scatterplot using ggplot2."
  • Make acceptance criteria very clear
    • False negatives are extremely frustrating

Starter Code

  • 0-10 lines (including blank lines)
  • More detailed than in a project
  • Use the imperative voice in comments

Wrong

# This sets plots to a nice size
options(repr.plot.width = 4)

# This loads the ggplot2 library
library(ggplot2)

Right

# Customize plot width
options(repr.plot.width = 4)

# Load ggplot2
library(ggplot2)

Hints

  • No more than 180-360 characters (30-60 words)
  • 1-4 hints per exercise
    • Usually map 1-to-1 with instructions
  • May include scaffolded code
  • "Assemble the data frame df using
    data.frame(t = t, x = sin(t), ___)"

Solutions

  • 1-15 lines
    • See guidelines for tab and bullet exercises below
  • Should run in 5 seconds or less on a regular laptop
  • So add datasets to the Docker image with requirements.sh unless it's completely impractical

Tab and Bullet Exercises

  • Shorter instructions are better
  • Shorter solutions are easier to understand
    • Use projects for long-form solutions
  • So use Tab and Bullet exercises for multi-step exercises
    • Explicit sub-goals improve learning
    • Words like "then" and "next" signal natural breaks
