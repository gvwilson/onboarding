Overview

  • Now understand goals for exercises and constraints on memory and cognition
  • See how different kinds of exercises achieve that
  • And how to implement those exercises using our platform

Requirements

  1. Individual exercises must be short (1-2 minutes)
    • People learn best in small steps
    • People drop out if exercises are longer
  2. Answers must be automatically gradable
    • Instant feedback is more engaging
    • Human inspection doesn't scale affordably

Engagement

Prefer coding exercises over non-coding exercises

  • People come to us for hands-on instruction
    • If they wanted a book, they'd buy a book
  • Applying new knowledge right away improves retention
    • Knowledge is more likely to be remembered
    • People are more likely to complete the course

Multiple Choice Exercise

Multiple Choice Exercise

What does the following fragment of Python print?

letters = 'abc'
for char in letters:
    print(char)
    letters.upper()
  1. A B C (doesn't understand instruction sequencing)
  2. a B C (thinks upper mutates its object)
  3. a b c (correct answer)
  4. error (thinks method results have to be assigned)

Normal Exercises

  • Write code that does X
    • But blank screens are intimidating
  • So give learners sample code
    • Makes auto-grading more reliable

Fill in the blanks so that the code below prints the string 'hat'. 

text = 'all that it is'
slice = text[____:____]
print(slice)

Minimal Changes

  • Correct a variable's initial value
  • Fix the bounds used to select a range of data
  • Fix the criteria used to select a subset of data
  • Replace a loop with a vectorized expression

Finding Bugs

  • Novices will spend most of their time doing this
    • Just like the rest of us
  • Give students a chance to practice another real-world skill
  • Exercise higher-level thinking
    • Reason from effect back to cause

Finding Bugs

subsum sums each sub-list of strictly increasing numbers: subsum([1, 3, 3, 4, 5, 1]) produces [4, 3, 9, 1]. Which bug does it contain?

  1. Every negative number starts a new sub-sequence.
  2. First value of each sub-sequence not in sub-sum.
  3. Last value of each sub-sequence not in sub-sum.
  4. Only re-starts sum when values actually decrease.

Tracing Execution

  • Given code, trace order in which lines are executed
  • Authentic task (comes up frequently in debugging)
  • Can be emulated using multiple choice exercise
    • Though this does increase cognitive load
  • Can also trace successive values

Tracing Execution

In what order do the lines in this code execute? 

A)     vals = [-1, 0, 1]
B)     inverse_sum = 0
       try:
            for v in vals:
C)             inverse_sum += 1/v
       except:
D)         pass
  1. A, B, C, C, C
  2. A, B, C, C, D
  3. A, B, C, C, D, C
